## Принципы документации DM-base

1. **Документация рядом с кодом.** Технические детали живут либо в `docs/*.md`, либо прямо рядом в README конкретного пакета (`frontend/README.md`, `netlify/README.md`).
2. **Single source of truth.** Любая инструкция (env vars, пайплайны, runbook’и) описывается в одном месте и линкуется отовсюду, вместо дублирования.
3. **Автоматически проверяемые примеры.** Кодовые сниппеты в документации должны быть копипастабельные и соответствовать существующим npm-скриптам.
4. **Release notes по функциям.** Каждая фича или architectural decision фиксируется в `docs/changelog.md` с датой, владельцем и влиянием на пользователя.
5. **UX-first описания.** Раздел «Для кого/зачем» идёт перед техническими деталями, чтобы onboard’ить нетехнических AppSec специалистов.
6. **Метки актуальности.** В начале документа указываем дату последнего апдейта и версию продукта, чтобы проще находить устаревшие сведения.
7. **Диаграммы как код.** Используем Mermaid/PlantUML в markdown, чтобы ревьюить архитектуру через Git diff.
